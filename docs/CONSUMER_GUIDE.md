# TANTRA Gated Bridge — Consumer Guide

**Version**: 1.0.0
**Date**: 2026-07-14

---

## 1. Overview

This guide is for systems and teams that consume TANTRA Gated Bridge services. TANTRA provides a zero-trust distributed execution pipeline. Consumers interact with TANTRA via HTTP APIs, filesystem reads, or Node.js module imports.

---

## 2. Consumer Types

| Consumer | Protocol | Purpose | Entry Point |
|---|---|---|---|
| API Client | HTTP POST | Submit workloads | `POST http://localhost:3000/initiate` |
| Observability Pipeline | File read | Ingest telemetry | `services/replay_persistence/data/replay_log.jsonl` |
| Monitoring System | HTTP GET | Health checks | `GET http://localhost:{port}/health` |
| Trace Analyst | Node.js require() | Reconstruct traces | `replay_reconstruction/reconstruction_tool.js` |
| Artifact Consumer | HTTP GET | Retrieve results | `GET http://localhost:3004/retrieve/:trace_id/:execution_id` |
| JWT Verifier | HTTP GET | Fetch public keys | `GET http://localhost:3001/.well-known/jwks.json` |
| Setu User | HTTP POST | End-user product | `POST http://localhost:8000/process` |

---

## 3. API Consumer

### Quick Start

```bash
# Submit a workload
curl -X POST http://localhost:3000/initiate \
  -H "Content-Type: application/json" \
  -d '{"workload": "my-task"}'
```

### Response

```json
{
  "trace_id": "a71bf018-cde6-4ef2-bafe-b11de9ecd68b",
  "execution_id": "c6e0eece-45d5-4cdc-bfc5-6c9c46ad4426",
  "cet_hash": "eeb8b5eaccc72d2514eb499e76c9957c490206cb2d2637114cd230faedf5a2bc",
  "status": "completed",
  "result": {
    "workload": "my-task",
    "output": "Processed my-task",
    "trace_id": "a71bf018-cde6-4ef2-bafe-b11de9ecd68b",
    "execution_id": "c6e0eece-45d5-4cdc-bfc5-6c9c46ad4426"
  },
  "artifact_location": "artifacts/a71bf018-.../c6e0eece-...",
  "duration_ms": 101
}
```

### Error Handling

| HTTP Code | Meaning | Action |
|---|---|---|
| 200 | Success | Process response |
| 400 | ID mutation detected | Do not modify trace_id/execution_id in requests |
| 401 | Unauthorized | Token invalid or replayed — obtain new token |
| 503 | Dependency unavailable | Retry with backoff; check service health |

### Important Notes

- `trace_id` and `execution_id` are generated by TANTRA — do not provide your own
- Each request is idempotent (same workload string may produce different trace IDs)
- Response times are typically 1-200ms
- Artifacts are stored in Bucket and retrievable via the artifact location

## 4. Observability Consumer

### Quick Start (Logstash)

```ruby
# logstash.conf
input {
  file {
    path => "/path/to/services/replay_persistence/data/replay_log.jsonl"
    codec => json_lines
  }
}
filter {
  if [event_type] =~ /^telemetry:/ {
    mutate { add_tag => ["tantra", "telemetry"] }
  }
}
output {
  elasticsearch { hosts => ["localhost:9200"] }
}
```

### Quick Start (Fluentd)

```xml
<source>
  @type tail
  path /path/to/services/replay_persistence/data/replay_log.jsonl
  pos_file /tmp/tantra.pos
  tag tantra.telemetry
  <parse>
    @type json
  </parse>
</source>
```

### Quick Start (Python)

```python
import json

with open("services/replay_persistence/data/replay_log.jsonl") as f:
    for line in f:
        event = json.loads(line.strip())
        if event.get("event_type", "").startswith("telemetry:"):
            print(f"[{event['service']}] {event['event_type']}: {event['status']}")
```

### Event Schema

See `services/observability/schema.json` for the full JSON Schema definition.

### Key Fields

| Field | Type | Description |
|---|---|---|
| `trace_id` | UUID | Execution trace identifier |
| `execution_id` | UUID | Individual execution identifier |
| `event_type` | string | `telemetry:<type>` |
| `service` | string | Originating service name |
| `status` | string | Event status |
| `payload.passive` | boolean | Always `true` |
| `timestamp` | ISO8601 | Event timestamp |
| `hash` | SHA-256 | Chain integrity hash |

## 6. Trace Analyst

### Quick Start

```javascript
const { reconstructTrace } = require('./services/replay_reconstruction/reconstruction_tool');

const trace = reconstructTrace('a71bf018-cde6-4ef2-bafe-b11de9ecd68b');
```

## 7. Artifact Consumer

### Quick Start

```bash
# Retrieve artifact
curl http://localhost:3004/retrieve/{trace_id}/{execution_id}
```

### Response

```json
{
  "trace_id": "uuid",
  "execution_id": "uuid",
  "result": {},
  "timestamp": "iso8601",
  "duration_ms": 123,
  "stored_at": "iso8601",
  "hash": "sha256-hex"
}
```

### Hash Verification

The `hash` field is a SHA-256 of the stored result. Verify integrity by recomputing:

```python
import hashlib, json
stored_hash = hashlib.sha256(json.dumps(result, sort_keys=True).encode()).hexdigest()
assert stored_hash == response["hash"]
```

## 8. JWT Verifier

### Quick Start

```bash
# Fetch JWKS
curl http://localhost:3001/.well-known/jwks.json
```

### Verify a Token

```javascript
const jwt = require('jsonwebtoken');
const axios = require('axios');

async function verifyToken(token) {
  const { data: jwks } = await axios.get('http://localhost:3001/.well-known/jwks.json');
  const decoded = jwt.decode(token, { complete: true });
  const kid = decoded.header.kid;
  const key = jwks.keys.find(k => k.kid === kid);

  // Convert JWK to PEM (use jose or jwk-to-pem library)
  const publicKey = jwkToPem(key);
  return jwt.verify(token, publicKey, { algorithms: [key.alg] });
}
```

## 9. Setu User

### Quick Start

```bash
# Submit via Setu
curl -X POST http://localhost:8000/process \
  -H "Content-Type: application/json" \
  -d '{"workload": "my-task"}'
```

### Flow

```
Setu → Core → Sarathi → Bridge → Execution → Bucket → Response
```

### Response

Setu returns the full TANTRA response with additional Setu metadata.
