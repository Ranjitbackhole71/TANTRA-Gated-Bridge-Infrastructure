## Known Limitations

1. **Replay cache is in-memory**: Bridge `replayCache` (`Map<jti, timestamp>`) lost on restart. Acknowledged in multiple audit documents.
2. **Workload execution is simulated**: `setTimeout` 100ms with hardcoded output. `execution_participant.js` adapter exists and IS configured (`EXECUTION_PARTICIPANT` env var set in `services/execution/.env`).
3. **InsightFlow remote not configured**: `INSIGHTFLOW_URL` and `INSIGHTFLOW_API_KEY` not set. Local telemetry works.
4. **No cross-reference between bucket artifacts and replay log**: Bucket has SHA-256 hash; replay log has separate SHA-256 chain. No cross-linking.
5. **Divergent directory copies**: `services/` and `tantra_gated_bridge/services/` differ (latter missing `execution_participant.js`, `insightflow/`, `observability/`).
6. **No CI/CD pipeline**: No automated test execution or deployment verification.
7. **No key rotation**: RSA key is static; no `kid` header; no JWKS endpoint for rotation.
8. **JWKS / kid not implemented**: Bridge fetches PEM from `/public-key`; no RFC 7517 JWKS support; JWT header has no `kid` claim.
9. **Algorithm locked to RS256**: EdDSA/Ed25519 not supported; no algorithm negotiation.
10. **Audience mismatch with Sarathi spec**: Bridge enforces `aud: tantra-bridge`; Sarathi requires `aud: bhiv-core-runtime`.
11. **cet_hash claim not validated**: Bridge checks `trace_id` and `execution_id` but does not enforce `cet_hash` claim continuity.
