## Key Directory Analysis: `services/sarathi/keys/`

### Are these production/private signing keys?
**YES.** They are RSA 2048-bit (`private.pem`) and Ed25519 (`ed25519_private.pem`) private keys used for JWT signing. The `key_meta.json` confirms RS256 and EdDSA algorithms with UUID key IDs and rotation tracking.

### Are they generated artifacts?
**YES.** The `.pem` files are generated by a key-generation or key-rotation process. The `.0.pem` files are previous-key backups from rotation.

### Were they previously tracked?
**YES.** All 9 files in `services/sarathi/keys/` were already tracked in git history (visible via `git ls-files --cached`). The staged changes represent key rotation — new "previous key" files and updated current key files.

### Should they be committed?
Generally **NO** — private keys in git are a security risk. However:
- These keys are **already tracked historically** in this repo
- The service code likely depends on these specific paths
- If these are development-only/demo keys (not production), the risk is lower
- The corresponding `tantra_gated_bridge/services/sarathi/keys/` directory is currently **untracked**, so the copy there is **not** being committed yet

**Recommendation:** Unstage private key files and add `*.pem` to `.gitignore`. If keys are required for the service to function, generate them at build/deploy time instead.

## SAFE_TO_COMMIT: NO

**Reasons:**
1. **Private keys are staged** — `private.pem`, `private.0.pem`, `ed25519_private.pem`, `ed25519_private.0.pem` contain actual private key material
